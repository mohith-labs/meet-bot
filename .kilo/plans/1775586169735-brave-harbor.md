# Fix: Transcriptions Not Showing / Same Transcription for Duplicate Meeting IDs

## Root Cause Analysis

### Bug 1: Same `nativeMeetingId` shows same transcript for different meetings

The entire transcript pipeline uses `platform + nativeMeetingId` (e.g., `google_meet/abc-defg-hij`) as the lookup key ŌĆö **not** the unique meeting UUID. When a user sends a bot to the same Google Meet room multiple times (recurring meetings, re-joining the same link), multiple meeting records share the same `nativeMeetingId`.

**The problem chain:**

1. Frontend meeting detail page (`/meetings/[id]`) receives the meeting **UUID** from the URL
2. It then fetches ALL meetings via `api.listMeetings()` to find the UUID ŌåÆ `nativeMeetingId` mapping (wasteful)
3. It calls `GET /transcripts/:platform/:nativeMeetingId` using the `nativeMeetingId`
4. The backend `TranscriptsService.getTranscript()` queries with `ORDER BY createdAt DESC` and returns only the **most recent** meeting for that `nativeMeetingId`
5. **Result**: Both older and newer meetings with the same `nativeMeetingId` display the newest meeting's transcript

**Affected files:**
- `backend/src/transcripts/transcripts.service.ts:22-25` ŌĆö always returns most recent meeting by `nativeMeetingId`
- `backend/src/transcripts/transcripts.controller.ts:23` ŌĆö route uses `/:platform/:nativeMeetingId`
- `frontend/src/app/(dashboard)/meetings/[id]/page.tsx:86-101` ŌĆö fetches all meetings, then uses `nativeMeetingId` to get transcript
- `backend/src/meetings/meetings.service.ts:26-29` ŌĆö `findByPlatformAndId` also returns most recent by `nativeMeetingId`

### Bug 2: Transcriptions sometimes not showing

Multiple contributing factors:

1. **Wrong meeting resolved**: As above, the `ORDER BY createdAt DESC` query may return a different meeting than the one being viewed, which may have no transcript segments yet
2. **Single retry only**: The auto-refresh (`meetings/[id]/page.tsx:117-124`) only triggers once when `status === 'completed' && segments.length === 0` ŌĆö if the flush to DB hasn't completed by then, transcripts are permanently missing from the UI until page reload
3. **Race condition**: Meeting status changes to `COMPLETED` before `saveBufferedTranscripts()` finishes writing to DB (the status update at `bots.service.ts:364-367` happens after the save, but the WebSocket broadcasts `COMPLETED` status immediately, causing the frontend to fetch before data is committed)

### Reference Implementation Comparison (recallai/google-meet-meeting-bot)

The reference repo avoids these issues by:
- Generating a **unique UUID per bot session** as the primary transcript key (`const meetingId = uuidv4()` in `runBot.ts:28`)
- Using `meetingId` as `@unique` on `MeetingTranscript` ŌĆö one transcript per bot session
- Using `@@unique([meetingId, start])` on `Segment` for dedup via `UPSERT`
- **All lookups by `meetingId` (UUID), never by Google Meet room code**

Our system already generates a unique UUID per meeting (`PrimaryGeneratedColumn('uuid')` in `meeting.entity.ts:30`). We just need to use it consistently in the API and frontend.

## Implementation Plan

### 1. Backend: Add UUID-based transcript endpoint

**File:** `backend/src/transcripts/transcripts.service.ts`

Add a new method `getTranscriptByMeetingId(userId, meetingId)`:
- Query meeting by UUID (`id`) instead of `nativeMeetingId`
- Verify the meeting belongs to the user
- Fetch transcript segments by `meetingId` (FK)
- Return same response shape as existing `getTranscript()`

**File:** `backend/src/transcripts/transcripts.controller.ts`

Add new route: `GET /transcripts/meeting/:meetingId`
- Uses UUID directly ŌĆö no ambiguity
- Place this route **before** the existing `:platform/:nativeMeetingId` route to avoid NestJS route conflicts

### 2. Backend: Add UUID-based single meeting detail endpoint

**File:** `backend/src/meetings/meetings.service.ts`

The `findById()` method exists (line 86-90) but doesn't include transcript segments. Update it to accept `userId` for auth and optionally load relations.

**File:** `backend/src/meetings/meetings.controller.ts`

Add new route: `GET /meetings/detail/:id`
- Fetches a single meeting by UUID with transcript segments
- Verifies the meeting belongs to the authenticated user
- Avoids the frontend needing to fetch ALL meetings to find one

### 3. Frontend: Update API client with new methods

**File:** `frontend/src/lib/api.ts`

Add two new methods:
- `getTranscriptByMeetingId(meetingId: string)` ŌåÆ calls `GET /transcripts/meeting/:meetingId`
- `getMeetingById(meetingId: string)` ŌåÆ calls `GET /meetings/detail/:meetingId`

### 4. Frontend: Fix meeting detail page to use UUID-based lookups

**File:** `frontend/src/app/(dashboard)/meetings/[id]/page.tsx`

Replace the current two-step approach:
```
// BEFORE (broken for duplicate nativeMeetingIds):
const allMeetings = await api.listMeetings();
const found = allMeetings.find((m) => m.id === meetingId);
const transcriptData = await api.getTranscript(found.platform, found.nativeMeetingId);

// AFTER (correct ŌĆö uses UUID directly):
const meetingDetail = await api.getMeetingById(meetingId);
const transcriptData = await api.getTranscriptByMeetingId(meetingId);
```

### 5. Frontend: Add retry/polling for transcript availability

**File:** `frontend/src/app/(dashboard)/meetings/[id]/page.tsx`

Replace the single-shot retry with a polling mechanism:
- When meeting status becomes `completed` and segments are empty, poll every 3 seconds (up to 5 attempts) for transcript data
- This handles the race condition where `COMPLETED` status is broadcast before transcript segments are fully flushed to DB

### 6. Keep existing `nativeMeetingId`-based endpoints (backward compatibility)

The existing `GET /transcripts/:platform/:nativeMeetingId` and `GET /meetings/:platform/:nativeMeetingId` endpoints remain unchanged for API key users who query by meeting code. No breaking changes.
